一、不同线程组配置计算预计压测时间方法：
1. Setup 线程组（循环次数控制）:
总耗时 = 启动延迟 + Ramp-up时间 + 循环次数 × 单次循环平均耗时
       = 5 + 2 + 1000 × T_loop
这种情况不能精确计算，输入框提示：Setup线程组通过循环次数控制，无法精确计算时间，请手动输入。
2、 标准 Thread Group（持续时间控制）：
总耗时 = 启动延迟 + Ramp-up时间 + 持续时间
       = 5 + 10 + 300 = 315 s
这里如果有使用循环次数，也无法计算精确时间，和上面一样的提示；
3、Stepping Thread Group（阶梯线程组）：
加压阶段步数：steps_up = ceil( (最大线程数 - step_start_users_count) / step_start_users_burst )
加压阶段总时长：T_up = steps_up × step_start_users_period
减压阶段步数：steps_down = ceil( step_start_users_count / step_stop_users_count )   // 或按实际从最大到0的阶梯
减压阶段总时长：T_down = steps_down × step_stop_users_period
总耗时 = step_initial_delay + T_up + step_flight_time + T_down
4. Ultimate Thread Group（终极线程组）：
单行的完整生命周期时长：T_row = initial_delay + startup_time + hold_load_for + shutdown_time
整个线程组总耗时（多行并行调度）：总耗时 = max( T_row1, T_row2, ... , T_rowN )
5、Setup + 其他线程组（混合）：
前置条件：setup/ThreadGroup线程组中没有循环次数，有同样参照1的提示；
执行顺序：默认顺序执行（如未勾选“独立运行每个线程组”）
总耗时 = T_setup + T_后续
6. 多个 Thread Group（多个标准线程组）
前置条件：setup/ThreadGroup线程组中没有循环次数，有同样参照1的提示；
总耗时 = Σ (每个线程组的启动延迟 + Ramp-up + 实际运行时长)

场景联调业务功能调整：
（1）、之前是点击联调后通过SSE流式推送检查项和结果，点击确定后开始联调，取参数配置中的少量线程更新jmx脚本后上传压力机试跑；这个流程比较繁杂，

（2）、勾选场景点击启动压测，则进入压测实时监控界面，实时显示进度和控制台输出，和之前逻辑不变；启动压测2个前置条件：① 状态联调通过待开始；② 预计耗时统计正确（主要循环次数时已设置）；
（3）、之前的联调执行接口业务逻辑先保留，后期试用后看不需要再去掉。

2、启动压测功能：
（1）、启动压测后需要更新的状态和处理的按钮操作权限：
    a. 更新压测场景状态为“进行中”；
    b. 场景列表和实时监控界面进度条实时更新进度；
    c. 文件管理中的jmx文件状态无需处理（减少业务逻辑依赖处理）；
    d. 进行中状态的压测场景主列表，操作按钮仅支持“强制停止”和“复制”；
    e. 进行中状态的压测场景子列表，当前压测开启的配置项操作按钮仅支持“复制”（状态开关禁用），其余关闭状态的子配置支持修改、删除（状态开关禁用且关闭状态）；
    注：也就是压测中不允许修改配置和场景数据，压测正常完成后会保留压测场景配置数据，便于后续对照测试报告分析。
（去掉使用中状态，sql初始化里面去掉即可，不需要新增迁移文件）；
（2）、启动压测后的业务操作：
    a、将最新配置更新到jmx脚本，MD5判断压力机已有文件或内容与当前一致则跳过上传，否则正常上传；
    上传到压力机后更新文件管理列表jmx分发状态（都是共享分发）和worker数量（都是1），备注：脚本已上传到单机(xxx)或者Master(xxx)；按钮变为清除；
    b、根据后台配置的启动命令启动压测；压测实时监控界面实时显示控制台日志输出；
（3）、压测中场景支持强制停止：kill JMeter 进程；更新场景状态 status=4（已取消）；停止日志收集；清理 Redis；
    - 取消场景支持“恢复”，点击后直接在linux启动压测命令（这里不需要更新jmx重新上传，和直接启动压测有区别的地方）；
    - 取消场景支持“删除”，删除后软删除场景和子场景，同时删除Linux中的jmx脚本、Minio中的jmx文件和关联的数据文件；
（4）压测完成后需要执行的业务操作：
    a、压测场景状态为“已完成”，场景主列表和子列表仅支持“复制”，不支持修改和删除，子列表状态按钮禁用。
    b、生成性能压测报告，需要收集的报告信息：Jmeter生成的report报告、Linux压力机的.jtl格式报告源文件、Linux各压力机的jmeter-xx.log日志文件；
    c、将报告上传到Minio存储。
（5）压测成功标志：
    情形一：log文件后50行有end of test标识且检查启动命令的PID进程已结束；
    情形二：log文件后50行没有结束标识，进程PID已结束且最后Active=0且Finished=目标总线程数（分布式时所有压力机线程数总和），
    也说明压测成功（实际压测过程中会有已结束单身没有end标识情况）；
（6）压测失败标志：
    情形一：没有生成jmeter.log日志文件且PID进程已结束；
    情形二：没有生成result.jtl结果文件且PID进程已结束；
    情形三：PID进程已结束，jmeter.log文件后50行Active!=0或Finished！=目标总线程数；
    情形四：PID进程已结束，但是实际压测时间（summary中有时间）远远小于预计耗时的一半。
（7）压测失败处理：主要包含压测启动失败、压测过程中断失败、压测报告收集失败等。
    a、上面任意场景触发，标记场景状态为“失败”；
    b、失败场景支持“恢复”，和取消场景的恢复逻辑一样；
    c、失败场景和子列表支持复制、修改和删除等常规操作，之前的针对不同线程组的操作逻辑不变；
    d、如果压测启动失败、压测过程中断失败，正常需要收集3类报告生成报告记录并上传；
